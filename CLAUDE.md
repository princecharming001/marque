# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

The app ships to users as **Yunicorn** (CFBundleDisplayName + all UI copy) but the repo, bundle id (`com.getmarque.app`), backend, and code all use the original name **Marque** — do not "fix" this mismatch. It's an iOS app that turns a creator's brand + one batch talking-head recording into many platform-ready short-form clips, learns from performance, and tightens the next batch.

Three deployable parts, each independently runnable:

- **`backend/`** — a single-file FastAPI server (`main.py`, ~4k lines) that holds every vendor key and is the app's only source of AI. Deployed on Render.
- **`ios/`** — the SwiftUI app (xcodegen-managed), talks only to the backend + Supabase (auth/storage).
- **`render/`** — a Remotion (React) video-render bridge deployed to AWS Lambda; the backend submits render plans to it.

> **The README and `docs/00`–`17` are an aspirational build-spec, not the implemented system.** They name vendors and modules that were never built that way (Shotstack, Ayrshare, Trigger.dev, R2, `LLMRouter`). The real stack is Anthropic + AssemblyAI + Apify + Supabase (Postgres/Auth/Storage) + Remotion-on-Lambda. Trust the code over the spec; `docs/DEPLOY.md` and `docs/ONBOARDING-DESIGN.md` are the docs that track reality.

## The one architectural rule: keyless-mock everywhere

Every AI/vendor path must return a sane deterministic result with **zero API keys set**, so the entire surface is testable and demoable offline. The pattern (see any endpoint in `main.py`):

```python
if not ANTHROPIC_KEY:            # or the relevant vendor key
    return {"mode": "mock", ...}  # deterministic stub
try:
    ... = extract_json(await anthropic(sys, usr, MODEL, tokens))
    return {"mode": "live", ...}
except HTTPException:
    return {"mode": "mock", ...}  # degrade, never 500
```

The whole test suite runs keyless and asserts the `mode: "mock"` behavior. **Any new endpoint that omits its keyless fallback is a bug** — it will 500 in the demo/CI build. Vendor keys are read from env at module load (`ANTHROPIC_API_KEY`, `APIFY_KEY`, `ASSEMBLYAI_KEY`, `PEXELS_KEY`, `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`, Remotion AWS keys).

## Commands

**Backend** (from `backend/`):
- `make test` or `python -m pytest -q` — full keyless suite (`test_main.py`, `test_editor_hardening.py`, `test_eval.py`). Currently ~271 tests.
- Run one test: `python -m pytest test_main.py::test_name -q`
- `make run` — local dev server on `:8000` (uvicorn --reload).
- Deploy is **manual** via the Render API — see `docs/DEPLOY.md` (the auto-deploy webhook is dead; service `srv-d94rk95ckfvc73ag4990`). Never assume a push deploys.

**iOS** (from repo root; drives the simulator, do not use raw xcodebuild):
- `./scripts/dev.sh build` — `xcodegen generate` + build + install on the pinned sim. Must end `BUILD SUCCEEDED`.
- `./scripts/dev.sh test [flow.yaml]` — build + install + run a Maestro flow (default `.maestro/flow-full.yaml`).
- **Sim gotcha:** the script pins one simulator UDID (`MARQUE_SIM`). If multiple sims are booted, `simctl install booted` can target the wrong one and you'll screenshot a stale app — shut down extras or pass the explicit UDID.
- `ios/project.yml` is the source of truth; `Marque.xcodeproj` is generated by xcodegen — edit the yml, never the pbxproj.

**Render bridge** (from `render/`):
- `npx tsc -p tsconfig.bridge.json` — typecheck the Node bridge (`lambda-render.ts`); this is the compile gate.
- `npm run deploy:site` — publish the Remotion site to Lambda (same serve URL, so no backend env change). Deploying render changes is a separate, gated step.

## Backend architecture (the part you'll touch most)

Single-file by design — resist splitting `main.py` without cause, but new subsystems may live in sibling modules imported into it (as `supabase_persistence.py` and `app/edl.py` already do).

- **`main.py`** — all FastAPI routes + the learning loop + the clip pipeline. Only async infra is per-request `asyncio.create_task`; **there is no cron/queue/worker**. `_lifespan` runs exactly one thing at boot: `_load_learning_state()`.
- **`prompts.py`** — every LLM prompt builder + the model constants (`OPUS`, `SONNET`, `HAIKU`) + content taxonomies (`STYLES`, `FORMAT_IDS`, `SIGNALS`, `NICHE_PRIORS`). Prompts live in git here, **not** in a remote flag service.
- **`supabase_persistence.py`** — durable storage via **raw PostgREST over httpx** (no ORM, no supabase SDK). Follow its `upsert_*/load_*` + `_*_COLS` filter pattern for any new table; every method catches its own errors and returns falsy (never raises into the hot path).
- **`app/edl.py`** — the video editor's core: an EDL (edit-decision-list) model, `apply_edl_ops`, and `build_render_plan` (the JSON contract the `render/` bridge consumes). EDL segments must stay monotonic; reordering happens only via `segment_order`.
- **`migrations.sql`** — schema; applied manually against Supabase (a live-DB change → a user-gated deploy).

**The learning loop** (the product's moat) is a 4-dimension Thompson bandit over `["pillar","style","format_id","hook_signal"]`:
`register_post` → `/v1/metrics/ingest` computes a reward `y` via `_compute_y(metrics, goal)` → `_update_arm` updates per-dimension `Beta(α,β)` → `_arms_for_prompt` shapes arms → `prompts.learning_block` injects them into script/hook/converse generation → `_calibration_signal` re-grounds predicted scores. Cold creators are seeded from `NICHE_PRIORS` (a `Beta(1,1)` default preserves the original math). Reward comes from **creator-confirmed metrics**, not public scrapes (a public scrape yields a degenerate constant reward — see the intel-port work).

**Clip pipeline:** upload (Supabase Storage) → AssemblyAI transcript → EDL generate→judge→repair (Anthropic) → `build_render_plan` → Remotion Lambda render → poll. Reels/brand-scan use Apify (`scrape_posts` takes a **handle**, not a post URL).

## Shipping to live (GitHub → Render / TestFlight)

**Standing rule first:** nothing reaches users automatically. `git push`, backend Render deploys, the Remotion site deploy, and iOS TestFlight builds are **explicit per-time user approvals** — implement locally, show the diff, and wait for an explicit "push it now." Local/feature-branch commits are fine. When on `main`, branch first (another session may be on `main` concurrently). Active intelligence work lives on branch `intel-port`.

### Where it lives on GitHub

**One monorepo holds everything:** `github.com/princecharming001/marque` (`git@github.com:princecharming001/marque.git`), default branch `main`. `backend/`, `ios/`, and `render/` are all subfolders of this single repo — there is **no separate repo**, and specifically **no website / landing-page / download-site repo** for Marque. The iOS app is distributed through **TestFlight / the App Store**, not a downloadable binary on a site, so there is nothing web-facing to deploy. (If you're thinking of a separate marketing/download site with its own repo, that belongs to a *different* project — it does not exist for Marque; don't go looking for one.)

### 1. Backend — the real "OTA" channel (Render)

Nearly all product behavior lives in `backend/`, so a backend deploy is how updates reach users without App Review. Full runbook: `docs/DEPLOY.md`.

- **Service:** `marque-api` on Render, id `srv-d94rk95ckfvc73ag4990`, Docker (repo-root `Dockerfile`), branch `main`, health check `/healthz`.
- **Live URL:** `https://marque-api.onrender.com`.
- **Auto-deploy is broken** — the GitHub repo has no Render webhook, so pushes do **not** deploy. You must trigger deploys manually via the Render API.

```bash
cd backend && python3 -m pytest -q && python3 -m eval.run_eval   # never push red
git push origin main                                             # (gated) push
curl -s -X POST "https://api.render.com/v1/services/srv-d94rk95ckfvc73ag4990/deploys" \
  -H "Authorization: Bearer $RENDER_API_KEY" -H "Content-Type: application/json" \
  --data '{"clearCache":"do_not_clear"}'                         # trigger deploy
# poll .../deploys/<dep-id> until "live", then:
curl -s https://marque-api.onrender.com/readyz                   # expect ai/scrape/publish: live
```

`RENDER_API_KEY` lives with the account owner (never committed). Deploy gotcha: `_clip_jobs` (clip/tweak edit sessions) are in-memory and **wiped on every deploy** — old clips show "edit session expired"; the learning stack in Supabase survives.

### 2. Render bridge (Remotion on Lambda)

Only when `render/` changes. `npx tsc -p tsconfig.bridge.json` (compile gate) → `npm run deploy:site` (publishes to the same Lambda serve URL, so no backend env change). Separate, separately-gated deploy.

### 3. iOS app (TestFlight / App Store)

Native SwiftUI → **no OTA for app code** (Apple prohibits it); any Swift/UI change needs a build. There is no archive/upload script — it's done via `xcodebuild archive` + `ios/ExportOptions.plist` (method `app-store-connect`, manual signing, profile "Marque App Store", Team `3TJ8RC3JCX`) → upload to App Store Connect (app id `6787590830`) via the ASC API key, then TestFlight. This is always a user-gated action. Bump `CURRENT_PROJECT_VERSION`/`CFBundleVersion` in `ios/project.yml` and remember to strip `MOCK_PAYWALL` before App Review.

## iOS notes

- SwiftUI, `@Observable` stores; features under `ios/Marque/Features/` (Onboarding, Home, Film, EditorView, RecordView, Chat, Paywall, etc.), backend calls behind `Adapters/BackendClient.swift`, state in `State/AppStore.swift`.
- `SWIFT_ACTIVE_COMPILATION_CONDITIONS` includes `MOCK_PAYWALL` — lets TestFlight mock-subscribe while no ASC IAP products exist. **Remove before App Review.**
- TestFlight signs headless via the ASC API key (Team `3TJ8RC3JCX`, app id `6787590830`, profile "Marque App Store"). Release config uses Manual signing with an explicit distribution profile.
- Fonts gotcha: a file named `Fraunces-Regular.ttf` is actually a Black cut — see the serif-key mapping in `App.tsx`/Info.plist if serif weights look wrong.
